# nanoagent 学习笔记

## 定位

`nanoagent` 不是工业级 Agent 平台，也不是一个可以直接照搬到复杂生产场景里的现成框架。它更像一套教学化的最小实现：每一章只增加少量代码，但每一层都清楚地回答了一个问题。

如果只保留一句话，我的判断是：

> 这套仓库真正讲清楚的，不是“怎么造一个很强的 Agent”，而是“一个能持续做事的 Agent，到底由哪些最小能力逐层长出来”。

## 按章节理解

### 01-essence

- 核心概念：`LLM + tools + loop`
- 关键价值：把 Agent 从“会回答”推进到“会执行并根据结果继续行动”
- 我的理解：这一章最重要的不是工具本身，而是反馈循环。没有循环，工具调用只是一次性动作；有了循环，模型才开始具备任务推进能力。

### 02-memory

- 核心概念：持久记忆与任务规划
- 关键价值：让 Agent 不再只是单轮响应器，而是开始具备延续目标的能力
- 我的理解：这里已经从“调用工具”进入“管理任务状态”。即使实现依然很轻，视角已经从 demo 迈向系统。

### 03-skills-mcp

- 核心概念：Rules、Skills、MCP
- 关键价值：把行为约束、能力扩展和外部工具接入分层表达出来
- 我的理解：这是整套仓库从“一个脚本”走向“可组合系统”的分水岭。真正有扩展性的 Agent，不只是 prompt 更长，而是规则、技能和工具接口被拆开了。

### 04-subagent

- 核心概念：子智能体委派
- 关键价值：把复杂任务拆给短生命周期、目标明确的执行单元
- 我的理解：这一层不是简单地“多开一个模型”，而是显式引入任务分工。它的工程意义在于隔离上下文、缩短局部决策链。

### 05-teams

- 核心概念：多智能体团队协作
- 关键价值：从一次性委派转向带角色和协作关系的持久编排
- 我的理解：`subagent` 更像临时工，`teams` 更像组织结构。这里真正引入的是身份、边界和协作关系，而不只是并行执行。

### 06-compact

- 核心概念：上下文压缩
- 关键价值：开始正面处理长任务场景下的上下文膨胀问题
- 我的理解：很多 Agent 示例都会忽略这一层，但它其实决定了系统能不能持续跑下去。没有上下文治理，再好的 loop 也会被窗口和成本拖垮。

### 07-safety

- 核心概念：命令过滤、用户确认、输出截断
- 关键价值：把“可执行”补全为“可控执行”
- 我的理解：这一章说明 Agent 系统不是能力越多越好，而是边界必须同步成长。安全不是附属件，而是基础件。

## 我怎么看 `full/agent-full.py`

`full/agent-full.py` 的价值不是“这就是最佳实践”，而是“把前七章的能力收拢成一张总图”。它更像一个教学总装版，方便建立完整心智模型，而不是建议直接上线的工程模板。

## 这套仓库最值得学的地方

- 用很小的代码量解释 Agent 的能力演进
- 每一层新增能力都能找到明确动机
- 章节边界清楚，适合做对照式学习
- 很适合拿来复盘“概念最小实现”

## 这套仓库的明显边界

- 状态管理依然偏轻
- 错误恢复和重试机制不够系统
- 权限模型与真实运行隔离较弱
- 可观测性、审计、回放能力不足
- 测试资产和当前文件树并不完全一致

## 结论

如果把这套仓库当教程，它是很好的材料；如果把它当产品骨架，它还远远不够。最适合它的位置，是“理解 Agent 结构的一阶样本”，而不是“直接投入复杂生产环境的现成框架”。
