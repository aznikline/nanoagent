# nanoAgent 学习笔记

## 总览

`nanoAgent` 不是“工业级 Agent 平台”，而是一套把 Agent 核心能力逐层拆开的教学仓库。它的长处不在于功能齐全，而在于把每一层能力都压到了非常低的理解门槛。

如果只抽一句话出来，我的结论是：

> 这套项目真正想教的不是“怎么造一个厉害的 Agent”，而是“Agent 到底由哪些最小部件组成，以及这些部件为什么会一层层长出来”。

## 分章理解

### 01-essence

- 核心概念：`LLM + tools + loop`
- 关键价值：把 Agent 从“会聊天”拉回到“能执行并根据结果继续行动”
- 我的理解：这一章最重要的不是工具本身，而是反馈循环。没有循环，工具调用只是一次性函数执行；有了循环，模型才开始表现出“任务推进”能力。

### 02-memory

- 核心概念：持久记忆与任务规划
- 关键价值：让 Agent 不再只是单轮响应器，而是开始具备延续目标的能力
- 我的理解：这里已经从“调用工具”进入“管理任务状态”。哪怕实现仍然很轻，思路已经从 demo 迈向系统。

### 03-skills-mcp

- 核心概念：Rules、Skills、MCP
- 关键价值：把 Agent 的行为配置、能力扩展和外部工具接入分层表达出来
- 我的理解：这是整套仓库从“一个脚本”走向“可组合系统”的分水岭。真正有扩展性的 Agent，靠的不只是 prompt，而是规则、技能和工具接口的拆分。

### 04-subagent

- 核心概念：子智能体委派
- 关键价值：把复杂任务拆给短生命周期、目标明确的执行单元
- 我的理解：这一层不是简单地“多开一个模型”，而是显式引入任务分工。它的工程意义在于隔离上下文、缩短决策链。

### 05-teams

- 核心概念：多智能体团队协作
- 关键价值：从一次性委派转向带角色和协作关系的持久编排
- 我的理解：`subagent` 更像临时工，`teams` 更像组织结构。这里真正引入的是身份、边界和协作关系，而不只是并行。

### 06-compact

- 核心概念：上下文压缩
- 关键价值：开始正面处理长任务下的上下文爆炸问题
- 我的理解：很多 Agent 示例都会忽略这一层，但它其实是能不能持续跑下去的关键。没有上下文治理，再好的 loop 最后也会被窗口和成本拖垮。

### 07-safety

- 核心概念：命令黑名单、用户确认、输出截断、Hook 管道
- 关键价值：把“可执行”补全为“可控执行”
- 我的理解：这章说明 Agent 系统不是能力越多越好，而是边界必须同步成长。安全能力不是附属件，而是基础件。

## 我对完整版本的理解

`full/agent-full.py` 的价值不是“这是最佳实践”，而是“把前七篇能力收拢成一张总图”。它更像一个教学总装版，便于建立完整心智模型，而不是建议直接上线的工程模板。

## 这套仓库最值得学的地方

- 用极小代码量解释 Agent 的能力演进
- 每一层新增能力都能找到明确动机
- 章节划分清楚，适合对照式学习
- 很适合拿来做“概念最小实现”复盘

## 这套仓库的边界

- 状态管理仍然偏轻
- 错误恢复和重试机制不够系统
- 权限模型和真实运行隔离比较弱
- 可观测性、审计、回放能力不足
- 测试资产和文件树存在不完全一致问题

## 我的总结

如果把这套仓库当教程，它是很好的材料；如果把它当产品骨架，它还远远不够。最适合它的位置，是“理解 Agent 结构的第一性原理样本”，而不是“直接投入复杂生产环境的现成框架”。
